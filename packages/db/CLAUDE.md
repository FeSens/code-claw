# Database — Drizzle ORM + PostgreSQL

## Structure

```
src/
├── schema/           Table definitions (one file per table)
│   ├── index.ts      Barrel export for all tables
│   └── users.ts      Users table
├── migrations/       Auto-generated by drizzle-kit — DO NOT EDIT
├── index.ts          Database client factory + re-exports
└── seed.ts           Seed script for development data
```

## Schema Design

Follow the established pattern in `src/schema/users.ts`:

```typescript
import { pgTable, text, timestamp, uuid } from "drizzle-orm/pg-core";

export const users = pgTable("users", {
  id: uuid("id").primaryKey().defaultRandom(),
  email: text("email").notNull().unique(),
  name: text("name").notNull(),
  createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
  updatedAt: timestamp("updated_at", { withTimezone: true })
    .notNull()
    .defaultNow()
    .$onUpdate(() => new Date()),
});
```

Rules:
- One file per table in `src/schema/`.
- Export the table from `src/schema/index.ts` (barrel file).
- Table names: `snake_case`, plural (`users`, `user_roles`, `order_items`).
- Column names: `snake_case` (`created_at`, `updated_at`, `user_id`).
- ALWAYS use `uuid` for primary keys with `.defaultRandom()`.
- ALWAYS add `createdAt` and `updatedAt` timestamps with timezone.
- Use `.$onUpdate(() => new Date())` on `updatedAt` for automatic updates.
- Use `.notNull()` by default. Only omit when the column is genuinely optional.
- Use `.unique()` for natural keys (email, slug, etc.).

## Relations

Define relations using Drizzle's `relations()` helper in the same file as the table:

```typescript
import { relations } from "drizzle-orm";

export const usersRelations = relations(users, ({ many }) => ({
  posts: many(posts),
}));
```

Rules:
- Relations go in the same file as the parent table.
- Export relations from `src/schema/index.ts` alongside tables.
- Use `one()` for belongs-to, `many()` for has-many.

## Migration Workflow

**This is critical. Follow this exact sequence:**

1. Edit schema files in `src/schema/`.
2. Run `pnpm db:generate` — Drizzle Kit generates a migration SQL file in `src/migrations/`.
3. **Review the generated SQL.** Read the migration file. Verify it does what you expect. Check for destructive operations (DROP TABLE, DROP COLUMN).
4. Run `pnpm db:migrate` — applies the migration to the database.
5. If the migration looks wrong, delete the generated file and re-run `pnpm db:generate` after fixing the schema.

**NEVER:**
- Edit migration files in `src/migrations/`. They are auto-generated.
- Run `pnpm db:push` in production. It's for development prototyping only.
- Run `pnpm db:migrate` without reviewing the generated SQL first.
- Add `DROP` statements manually. Let Drizzle Kit generate them.

**Destructive changes (renaming/removing columns):**
- Drizzle Kit may generate a `DROP COLUMN` + `ADD COLUMN` instead of `ALTER COLUMN RENAME`. Always review.
- For column renames: create a new migration that adds the new column, copies data, drops the old column. Do NOT rely on auto-generation for renames.

## Database Client

The client is created in `src/index.ts`:

```typescript
import { drizzle } from "drizzle-orm/postgres-js";
import postgres from "postgres";
import * as schema from "./schema/index.js";

export function createDb(connectionString: string) {
  const client = postgres(connectionString);
  return drizzle(client, { schema });
}

export type Database = ReturnType<typeof createDb>;
```

Rules:
- Import `createDb` in consumers. NEVER create a database client directly in app code.
- The `schema` parameter enables Drizzle's relational query API (`db.query.users.findMany()`).
- NEVER import `postgres` directly in app code. Only through `createDb`.

## Seeding

The seed script (`src/seed.ts`) inserts development data:

```typescript
const db = createDb(DATABASE_URL);

async function seed() {
  await db.insert(users).values([
    { email: "alice@example.com", name: "Alice" },
  ]);
}
```

Rules:
- Seed data should be idempotent. Use `ON CONFLICT DO NOTHING` or check for existing data.
- Seed script runs via `pnpm db:seed`.
- NEVER put real user data or secrets in seed files.
- Keep seed data minimal — just enough for development and E2E tests.

## Drizzle Config

`drizzle.config.ts` points to the schema and migrations directories:

```typescript
export default defineConfig({
  schema: "./src/schema/index.ts",
  out: "./src/migrations",
  dialect: "postgresql",
  dbCredentials: { url: process.env.DATABASE_URL! },
});
```

This file is NOT in the read-only harness list — you may need to update it when adding new schema files. But only change `schema` and `out` paths, not the dialect or credentials pattern.

## Common Mistakes

- NEVER use raw SQL (`db.execute(sql\`...\`)`) when Drizzle's query builder can do it. Raw SQL bypasses type safety.
- NEVER store passwords in plain text. Use bcrypt or argon2 (handled in the service layer, not the schema).
- NEVER make a column nullable "just in case." Start strict, loosen later if needed.
- NEVER forget to export new tables from `src/schema/index.ts`. If it's not exported, Drizzle won't see it.
